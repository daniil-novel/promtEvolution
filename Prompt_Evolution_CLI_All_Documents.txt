# Документация проекта: CLI-утилита для эволюционного улучшения промптов по методу SCOPE

## Общая информация

**Название проекта:** Prompt Evolution CLI  
**Тип проекта:** CLI-утилита на Python  
**Основной интерфейс:** терминал VS Code  
**Назначение:** генерация, улучшение, тестирование и выбор лучшего промпта по методу SCOPE  
**LLM-провайдеры:** OpenRouter через API key; GigaChat через отдельную ручку подключения  
**Основные метрики:** pass@1, pass@k  
**Базовое количество тесткейсов:** 40

# BRD — Business Requirements Document

## 1.1. Назначение документа

BRD описывает, зачем нужен проект, какую бизнес-проблему он решает, кто будет пользоваться CLI-утилитой, какие цели должны быть достигнуты и какие ограничения необходимо учитывать.

## 1.2. Краткое описание проекта

Prompt Evolution CLI — это консольная Python-утилита, которая запускается из терминала VS Code и помогает автоматически создавать или улучшать промпты.

Пользователь передаёт в утилиту описание задачи. Дополнительно он может передать уже существующий промпт и набор тесткейсов. Если промпта нет, утилита должна сгенерировать несколько вариантов промпта. Если тесткейсов нет или их меньше нужного количества, утилита должна сгенерировать недостающие тесткейсы отдельным LLM-вызовом.

После этого программа прогоняет промпты по тесткейсам, считает метрики pass@1 и pass@k, выбирает лучший вариант, анализирует ошибки и улучшает промпт по методу SCOPE.

## 1.3. Бизнес-проблема

Создание хороших промптов сейчас часто происходит вручную:

1. Пользователь пишет промпт.
2. Запускает его на нескольких примерах.
3. Смотрит, что не работает.
4. Редактирует промпт вручную.
5. Повторяет процесс много раз.

У такого подхода есть проблемы:

- нет объективной оценки качества промпта;
- нет стабильного набора тесткейсов;
- сложно понять, какой промпт лучше;
- сложно проверить, что новый промпт не сломал старые сценарии;
- ручная доработка занимает много времени;
- качество зависит от опыта prompt engineer;
- промпт может хорошо работать на одном примере, но проваливаться на других.

## 1.4. Цель проекта

Создать CLI-утилиту, которая автоматизирует процесс prompt engineering:

```text
описание задачи → генерация тестов → генерация/улучшение промптов → прогон тестов → подсчёт метрик → выбор лучшего промпта → улучшение по SCOPE
```

Главная цель — превратить работу с промптами из ручного подбора в измеримый инженерный процесс.

## 1.5. Пользователи

| Пользователь | Зачем использует утилиту |
|---|---|
| Prompt engineer | Быстро создавать и улучшать промпты |
| Разработчик | Проверять промпты для AI-функций в проекте |
| QA-инженер | Тестировать промпты на наборах кейсов |
| AI engineer | Сравнивать разные версии промптов по метрикам |
| Студент/исследователь | Экспериментировать с SCOPE и prompt evolution |

## 1.6. Основная бизнес-ценность

Утилита должна дать пользователю возможность:

1. Создать рабочий промпт с нуля.
2. Улучшить уже существующий промпт.
3. Автоматически получить тесткейсы для проверки.
4. Проверить промпт не на одном примере, а на наборе тестов.
5. Посчитать pass@1 и pass@k.
6. Сравнить несколько промптов между собой.
7. Получить итоговый лучший промпт.
8. Получить Markdown-отчёт с результатами.
9. Повторять улучшение заданное количество итераций.
10. Использовать OpenRouter или GigaChat как LLM-провайдера.

## 1.8. Бизнес-ограничения

| Ограничение | Описание |
|---|---|
| Стоимость LLM-вызовов | Чем больше тестов, итераций и кандидатов, тем дороже запуск |
| Недетерминированность LLM | Одинаковые запуски могут давать разные результаты |
| Качество тесткейсов | Если тесты плохие, метрики будут бесполезны |
| Ограничения API | Возможны rate limit, ошибки авторизации, лимиты контекста |
| Зависимость от провайдера | OpenRouter и GigaChat могут иметь разные API, модели и форматы ответов |
| Безопасность ключей | API key нельзя хранить в коде и коммитить в Git |

## 1.9. Критерии успеха

3. Сгенерировать промпт при отсутствии исходного промпта.
4. Улучшить переданный промпт.
5. Сгенерировать тесткейсы при их отсутствии.
6. Догенерировать тесткейсы до заданного количества, например до 40.
7. Прогнать несколько промптов по тесткейсам.
8. Посчитать pass@1 и pass@k.
9. Выбрать лучший промпт.
10. Сохранить итоговый промпт в файл.
11. Сохранить отчёт в Markdown.
12. Работать через OpenRouter API key.
13. Иметь предусмотренное место для подключения GigaChat.

# SRS — Software Requirements Specification

## 2.1. Назначение документа

SRS описывает, что именно должна делать CLI-утилита: какие входные данные принимает, какие команды поддерживает, какие сценарии выполняет, какие метрики считает и какие файлы создаёт.

## 2.2. Общее описание системы

Prompt Evolution CLI — это Python-приложение, запускаемое из командной строки.

Минимальный пользовательский сценарий:

```bash
prompt-evolve run --task task.md
```

Расширенный сценарий:

```bash
prompt-evolve run \
  --task task.md \
  --prompt prompt.md \
  --tests tests.json \
  --provider openrouter \
  --model openai/gpt-4.1 \
  --target-tests 40 \
  --iterations 3 \
  --candidates 4 \
  --pass-k 4 \
  --out runs/run_001
```

## 2.3. Основной принцип работы

Утилита должна работать по следующему пайплайну:

```text
1. Прочитать описание задачи.
2. Прочитать исходный промпт, если он передан.
3. Прочитать тесткейсы, если они переданы.
4. Если тесткейсов нет — сгенерировать их.
5. Если тесткейсов меньше нужного количества — догенерировать недостающие.
6. Если промпта нет — сгенерировать несколько промптов-кандидатов.
7. Если промпт есть — сгенерировать несколько улучшенных вариантов.
8. Прогнать каждый промпт-кандидат по тесткейсам.
9. Оценить ответы.
10. Посчитать pass@1 и pass@k.
11. Выбрать лучший промпт.
12. Проанализировать ошибки.
13. Сгенерировать SCOPE-guidelines.
14. Обновить промпт.
15. Повторить цикл заданное количество итераций.
16. Сохранить итоговый промпт и отчёт.
```

## 2.4. Поддерживаемые команды CLI

## 2.4.1. Команда `run`

Основная команда полного запуска.

```bash
prompt-evolve run --task task.md
```

Команда должна:

1. Выполнить полный цикл генерации/улучшения промпта.
2. Сгенерировать или загрузить тесткейсы.
3. Провести оценку промптов.
4. Сохранить результаты.

## 2.4.2. Команда `generate-tests`

Команда только для генерации тесткейсов.

```bash
prompt-evolve generate-tests \
  --task task.md \
  --target-tests 40 \
  --out tests.json
```

Команда должна создать файл с тесткейсами без запуска prompt evolution.

## 2.4.3. Команда `evaluate`

Команда для проверки одного промпта на готовом наборе тестов.

```bash
prompt-evolve evaluate \
  --task task.md \
  --prompt prompt.md \
  --tests tests.json
```

Команда должна посчитать метрики для одного промпта.

## 2.4.4. Команда `init-config`

Команда для создания шаблона конфигурации.

```bash
prompt-evolve init-config --out prompt-evolve.yaml
```

Команда должна создать пример конфигурационного файла.

## 2.5. CLI-параметры команды `run`

| Параметр | Обязательный | Значение по умолчанию | Описание |
|---|---:|---:|---|
| `--task` | Да | — | Путь к Markdown-файлу с описанием задачи |
| `--prompt` | Нет | `None` | Путь к исходному промпту |
| `--tests` | Нет | `None` | Путь к JSON-файлу с тесткейсами |
| `--provider` | Нет | `openrouter` | LLM-провайдер: `openrouter` или `gigachat` |
| `--model` | Нет | зависит от провайдера | Название модели |
| `--target-tests` | Нет | `40` | Сколько тесткейсов должно быть в итоговом наборе |
| `--iterations` | Нет | `3` | Количество итераций улучшения |
| `--candidates` | Нет | `4` | Количество промптов-кандидатов на итерацию |
| `--pass-k` | Нет | `4` | Значение k для pass@k |
| `--user-tests-only` | Нет | `false` | Использовать только переданные тесты и ничего не догенерировать |
| `--reasoning` | Нет | `max` | Уровень reasoning для генерации тестов и self-check |
| `--self-check` | Нет | `true` | Включить self-check |
| `--out` | Нет | `runs/latest` | Папка для результатов |
| `--config` | Нет | `None` | Путь к YAML-конфигу |

## 2.6. Конфигурационный файл

Утилита должна поддерживать YAML-конфиг.

Пример `prompt-evolve.yaml`:

```yaml
provider: openrouter
model: openai/gpt-4.1

target_tests: 40
iterations: 3
candidates: 4
pass_k: 4
reasoning: max
self_check: true
user_tests_only: false

output:
  dir: runs/latest
  save_markdown_report: true
  save_json_report: true
  save_final_prompt: true

openrouter:
  api_key_env: OPENROUTER_API_KEY
  base_url: https://openrouter.ai/api/v1

gigachat:
  enabled: false
  credentials_env: GIGACHAT_CREDENTIALS
  base_url: null
```

CLI-параметры должны иметь приоритет над параметрами из конфига.

## 2.7. Подключение LLM-провайдеров

## 2.7.1. OpenRouter

Для OpenRouter утилита должна использовать API key из переменной окружения.

Пример `.env`:

```bash
OPENROUTER_API_KEY=your_openrouter_api_key_here
```

Пример запуска:

```bash
prompt-evolve run \
  --task task.md \
  --provider openrouter \
  --model openai/gpt-4.1
```

Требования:

1. API key не должен храниться в коде.
2. API key не должен выводиться в логах.
3. При отсутствии ключа утилита должна вывести понятную ошибку.
4. Должна быть возможность указать модель через `--model`.
5. Должна быть возможность указать base URL через конфиг.

## 2.7.2. GigaChat

В проекте обязательно должна быть предусмотрена ручка подключения GigaChat.

Под ручкой понимается отдельный провайдерный интерфейс, через который можно подключить GigaChat без переписывания основной логики CLI.

Пример запуска:

```bash
prompt-evolve run \
  --task task.md \
  --provider gigachat \
  --model GigaChat-Pro
```

Требования:

1. В CLI должен быть параметр `--provider gigachat`.
2. В конфиге должен быть отдельный блок `gigachat`.
3. Ключи/credentials должны передаваться через переменные окружения.
4. Основной пайплайн не должен зависеть от конкретного провайдера.
5. При незаполненной конфигурации GigaChat утилита должна сообщить, что провайдер не настроен.
6. Подключение GigaChat должно быть реализовано через отдельный клиент или адаптер.

## 2.8. Входные файлы

## 2.8.1. Файл задачи

Файл `task.md` должен содержать описание задачи, для которой нужен промпт.

Пример:

```markdown
# Задача

Нужно создать системный промпт для AI-агента, который превращает голосовую расшифровку пользователя в аккуратное Telegram-сообщение.

# Требования

- Сохранять авторский смысл.
- Убирать повторы.
- Не искажать речь.
- Форматировать текст под Telegram Markdown.
- Делить текст на логические блоки.
```

## 2.8.2. Файл исходного промпта

Файл `prompt.md` содержит промпт, который нужно улучшить.

Если файл не передан, утилита должна создать промпт с нуля.

## 2.8.3. Файл тесткейсов

Файл `tests.json` содержит пользовательские тесткейсы.

Если файл не передан, утилита должна сгенерировать тесткейсы самостоятельно.

Если тесткейсов меньше `--target-tests`, утилита должна по умолчанию догенерировать недостающие.

Если пользователь передал `--user-tests-only`, догенерация не выполняется.

## 2.9. Формат тесткейса

Тесткейсы должны храниться в JSON.

```json
[
  {
    "id": "TC-001",
    "name": "Базовый happy path",
    "type": "happy_path",
    "priority": "high",
    "input": "Сырой пользовательский запрос или пример входа",
    "expected_behavior": "Что должен сделать промпт",
    "evaluation_criteria": [
      "Ответ должен сохранить смысл входного текста",
      "Ответ должен соответствовать требуемому формату",
      "Ответ не должен добавлять факты от себя"
    ]
  }
]
```

Обязательные поля:

1. `id`.
2. `name`.
3. `type`.
4. `priority`.
5. `input`.
6. `expected_behavior`.
7. `evaluation_criteria`.

## 2.10. Генерация тесткейсов

Если тесткейсы не переданы, утилита должна сгенерировать их отдельным LLM-вызовом.

Если тесткейсы переданы, но их меньше нужного количества, утилита должна догенерировать недостающие.

По умолчанию итоговое количество тесткейсов — 40.

Распределение тесткейсов по умолчанию:

| Тип тесткейса | Количество для 40 тестов |
|---|---:|
| Happy path | 12 |
| Сложные реалистичные кейсы | 8 |
| Edge cases | 7 |
| Negative cases | 5 |
| Format compliance | 4 |
| Adversarial cases | 2 |
| Regression cases | 2 |

## 2.11. Self-check regulator

Self-check regulator — это дополнительная проверка качества.

Он должен использоваться:

1. После генерации тесткейсов.
2. После оценки ответов.
3. После генерации SCOPE-guidelines.

Self-check должен проверять:

- нет ли дублей;
- не слишком ли общий тест;
- проверяет ли тест именно нужную задачу;
- есть ли понятные критерии оценки;
- не противоречит ли тест другим тестам;
- корректно ли evaluator оценил ответ;
- не добавила ли guideline лишнее или вредное правило.

## 2.12. Генерация и улучшение промптов

Если исходный промпт отсутствует, утилита должна сгенерировать несколько промптов-кандидатов.

Если исходный промпт передан, утилита должна использовать его как baseline и создать несколько улучшенных версий.

Каждый промпт-кандидат должен содержать:

1. Роль агента.
2. Цель агента.
3. Правила обработки входа.
4. Ограничения.
5. Формат ответа.
6. Правила самопроверки.
7. Запрет на галлюцинации.
8. Требования к качеству результата.

## 2.13. Оценка промптов

Утилита должна прогонять каждый промпт-кандидат по каждому тесткейсу.

Для каждого запуска сохраняются:

1. ID тесткейса.
2. ID промпта-кандидата.
3. Вход теста.
4. Ответ модели.
5. Результат evaluation.
6. Причина pass/fail.
7. Тип ошибки, если тест провален.

## 2.14. Evaluator

Evaluator должен получать:

1. Тесткейс.
2. Ответ модели.
3. Критерии оценки.

Evaluator должен возвращать:

```json
{
  "test_case_id": "TC-001",
  "passed": true,
  "score": 0.94,
  "reason": "Ответ соответствует всем критериям.",
  "failed_criteria": [],
  "error_type": null
}
```

Если тест провален:

```json
{
  "test_case_id": "TC-008",
  "passed": false,
  "score": 0.41,
  "reason": "Ответ нарушил требуемый формат Markdown.",
  "failed_criteria": [
    "Ответ должен быть в Markdown"
  ],
  "error_type": "format_violation"
}
```

## 2.15. Метрики

## 2.15.1. pass@1

pass@1 показывает, какую долю тесткейсов прошёл лучший одиночный промпт.

```text
pass@1 = passed_tests / total_tests
```

Пример:

```text
34 / 40 = 0.85
```

## 2.15.2. pass@k

pass@k показывает, какую долю тесткейсов прошёл хотя бы один из k промптов-кандидатов.

```text
pass@k = tests_passed_by_at_least_one_of_k_candidates / total_tests
```

Пример:

```text
Если из 40 тестов хотя бы один из 4 кандидатов прошёл 38 тестов, pass@4 = 38 / 40 = 0.95
```

## 2.15.3. Дополнительные метрики

Утилита также может считать:

| Метрика | Описание |
|---|---|
| `format_pass_rate` | Доля ответов в правильном формате |
| `critical_pass_rate` | Доля пройденных critical/high тестов |
| `average_score` | Средний score evaluator |
| `improvement_delta` | Разница между baseline и final prompt |
| `failed_tests_count` | Количество проваленных тестов |
| `token_usage` | Расход токенов |
| `estimated_cost` | Примерная стоимость запуска |

## 2.16. Выбор лучшего промпта

Лучший промпт выбирается по следующим приоритетам:

1. Максимальный pass@1.
2. Максимальный critical_pass_rate.
3. Максимальный average_score.
4. Минимальное количество format violations.
5. Меньшая длина промпта.
6. Меньшая стоимость выполнения.

Если исходный baseline prompt оказался лучше всех кандидатов, утилита должна оставить baseline.

## 2.17. SCOPE-улучшение

После каждого прогона утилита должна анализировать ошибки и создавать правила улучшения промпта.

Типы правил:

| Тип | Описание |
|---|---|
| Corrective guideline | Исправляет конкретную ошибку |
| Enhancement guideline | Улучшает поведение, даже если явной ошибки не было |

Пример corrective guideline:

```text
When the required output format is Markdown, validate the final answer structure before returning it.
```

Пример enhancement guideline:

```text
When the task has multiple possible user inputs, explicitly handle happy path, edge cases and malformed inputs.
```

Guidelines должны быть:

1. Короткими.
2. Конкретными.
3. Применимыми.
4. Не дублирующими уже существующие правила.
5. Полезными для повышения pass@1/pass@k.

## 2.18. Итерации

Пользователь должен иметь возможность задать количество итераций.

Пример:

```bash
prompt-evolve run --task task.md --iterations 5
```

Одна итерация включает:

1. Генерацию или улучшение промптов-кандидатов.
2. Прогон по тесткейсам.
3. Оценку.
4. Подсчёт метрик.
5. Выбор лучшего промпта.
6. Анализ ошибок.
7. Генерацию guidelines.
8. Обновление промпта.

## 2.19. Выходные файлы

После запуска утилита должна создать папку результата.

Пример:

```text
runs/run_001/
  final_prompt.md
  report.md
  report.json
  tests.json
  candidates/
    iteration_1_candidate_1.md
```

| `report.md` | Человекочитаемый отчёт |
| `report.json` | Машиночитаемый отчёт |
| `tests.json` | Итоговый набор тесткейсов |
| `run.log` | Лог запуска |

# FRD — Functional Requirements Document

## 3.1. Назначение документа

FRD описывает конкретную функциональность CLI-утилиты: команды, действия пользователя, правила валидации, статусы запуска, форматы файлов и ожидаемое поведение.

## 3.2. Пользовательский сценарий 1: создать промпт с нуля

### Описание

Пользователь передаёт только описание задачи. Утилита сама создаёт тесткейсы, генерирует промпты и выбирает лучший.

### Команда

```bash
prompt-evolve run --task task.md
```

### Ожидаемое поведение

1. Утилита читает `task.md`.
2. Проверяет наличие OpenRouter API key или настроенного провайдера.
3. Генерирует 40 тесткейсов.
4. Проверяет тесткейсы через self-check.
5. Генерирует 4 промпта-кандидата.
6. Прогоняет кандидатов по тестам.
7. Считает pass@1 и pass@k.
8. Выбирает лучший промпт.
9. Выполняет 3 итерации улучшения.
10. Сохраняет `final_prompt.md` и `report.md`.

## 3.3. Пользовательский сценарий 2: улучшить существующий промпт

### Команда

```bash
prompt-evolve run --task task.md --prompt prompt.md
```

### Ожидаемое поведение

1. Утилита использует `prompt.md` как baseline.
2. Генерирует тесткейсы, если они не переданы.
3. Считает baseline pass@1.
4. Создаёт улучшенные версии промпта.
5. Сравнивает их с baseline.
6. Выбирает лучший вариант.
7. Если baseline лучше, оставляет baseline.
8. Сохраняет отчёт с объяснением выбора.

## 3.4. Пользовательский сценарий 3: использовать свои тесткейсы

### Команда

```bash
prompt-evolve run \
  --task task.md \
  --prompt prompt.md \
  --tests tests.json
```

### Ожидаемое поведение

1. Утилита загружает пользовательские тесткейсы.
2. Валидирует их структуру.
3. Если тестов меньше 40, догенерирует недостающие.
4. Если указан `--user-tests-only`, использует только пользовательские тесты.
5. Запускает оценку промптов.

## 3.5. Пользовательский сценарий 4: запуск через GigaChat

### Команда

```bash
prompt-evolve run \
  --task task.md \
  --provider gigachat \
  --model GigaChat-Pro
```

### Ожидаемое поведение

1. Утилита выбирает GigaChat-провайдер.
2. Проверяет наличие GigaChat credentials в переменной окружения.
3. Если credentials отсутствуют, выводит понятную ошибку.
4. Если GigaChat настроен, выполняет тот же pipeline, что и для OpenRouter.
5. Остальная логика не меняется.

## 3.6. Пользовательский сценарий 5: только генерация тесткейсов

### Команда

```bash
prompt-evolve generate-tests --task task.md --target-tests 40 --out tests.json
```

### Ожидаемое поведение

1. Утилита читает описание задачи.
2. Генерирует 40 тесткейсов.
3. Проверяет их через self-check.
4. Сохраняет `tests.json`.
5. Не запускает prompt evolution.

## 3.7. Пользовательский сценарий 6: только оценка одного промпта

### Команда

```bash
prompt-evolve evaluate \
  --task task.md \
  --prompt prompt.md \
  --tests tests.json
```

### Ожидаемое поведение

1. Утилита загружает task, prompt и tests.
2. Прогоняет prompt по всем тестам.
3. Считает pass@1.
4. Формирует отчёт.
5. Не генерирует новые промпты.

## 3.8. Валидация входных данных

| Данные | Правило | Ошибка |
|---|---|---|
| `task.md` | Файл существует и не пустой | `Task file is missing or empty` |
| `prompt.md` | Если передан, файл существует и не пустой | `Prompt file is missing or empty` |
| `tests.json` | Если передан, содержит список тесткейсов | `Invalid tests file format` |
| `--target-tests` | Число от 1 до 500 | `target-tests must be between 1 and 500` |
| `--iterations` | Число от 1 до 50 | `iterations must be between 1 and 50` |
| `--candidates` | Число от 1 до 20 | `candidates must be between 1 and 20` |
| `--pass-k` | Не больше `--candidates` | `pass-k cannot be greater than candidates` |
| `--provider` | `openrouter` или `gigachat` | `Unsupported provider` |

## 3.9. Статусы запуска

Во время выполнения утилита должна показывать понятные статусы в терминале.

```text
[1/9] Loading input files...
[2/9] Checking provider configuration...
[3/9] Generating test cases...
[4/9] Running self-check for test cases...
[5/9] Generating prompt candidates...
[6/9] Evaluating candidates...
[7/9] Calculating metrics...
[8/9] Applying SCOPE improvements...
[9/9] Saving final report...
```

## 3.10. Ошибки и поведение

| Ситуация | Поведение |
|---|---|
| Нет API key OpenRouter | Вывести ошибку и инструкцию добавить `OPENROUTER_API_KEY` |
| GigaChat не настроен | Вывести ошибку, что провайдер выбран, но credentials отсутствуют |
| LLM вернула невалидный JSON | Повторить запрос со строгой инструкцией вернуть JSON |
| Не удалось сгенерировать тесты | Завершить запуск и сохранить partial log |
| Все промпты провалили тесты | Выбрать лучший по score и явно указать, что качество низкое |
| Превышен лимит контекста | Уменьшить размер кандидата или применить сжатие prompt |
| Превышен лимит стоимости | Остановить запуск и сохранить частичный отчёт |

# Краткое итоговое summary

Проект должен быть не web-сервисом и не сложной платформой, а простой CLI-утилитой на Python, которую можно запускать из терминала VS Code.

Основной сценарий:

```bash
prompt-evolve run --task task.md --provider openrouter --target-tests 40 --iterations 3
```

Утилита должна:

1. Принять описание задачи.
2. Принять или сгенерировать промпт.
3. Принять или сгенерировать тесткейсы.
4. Использовать отдельный LLM-вызов для тесткейсов.
5. Использовать self-check regulator.
6. Сгенерировать несколько промптов-кандидатов.
7. Прогнать их по тестам.
8. Посчитать pass@1 и pass@k.
9. Выбрать лучший промпт.
10. Улучшить его по SCOPE.
11. Повторить процесс заданное количество итераций.
12. Сохранить `final_prompt.md`, `report.md`, `report.json`, `tests.json`.

Главное техническое ограничение:

```text
Это должна быть CLI-утилита, а не web-приложение.
```

Главное интеграционное требование:

```text
OpenRouter должен работать через API key.
GigaChat должен иметь отдельную ручку подключения через provider interface.
```
